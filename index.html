Функціональні вимоги

ID Пріоритет Назва вимоги Опис вимоги (Критерії перевірки)
FR-01 High Пошук креаторів Система повинна повертати список профілів креаторів, що відповідають введеному текстовому запиту користувача, протягом не більше 2 секунд.
FR-02 High Фільтрація результатів Система повинна відображати лише ті профілі або роботи, які відповідають вибраним користувачем параметрам (категорія, теги, рейтинг).
FR-03 High Реєстрація облікового запису Система повинна створювати новий запис користувача в базі даних після заповнення форми (email, пароль, ім'я) та підтвердження унікальності email.
FR-04 High Авторизація користувача Система повинна надавати доступ до закритих функцій профілю після введення коректної пари логін/пароль та видавати токен сесії.
FR-05 Medium Додавання робіт до портфоліо Система повинна дозволяти авторизованому креатору завантажувати медіа-файл (JPG/PNG) з обов’язковим заповненням полів "Назва" та "Опис".
FR-06 High Блокування користувачів Система повинна дозволяти адміністратору змінювати статус облікового запису на "Заблокований", що миттєво припиняє доступ користувача до системи.
FR-07 Medium Редагування контенту Система повинна зберігати зміни в описі або зображеннях роботи після натискання кнопки "Зберегти" та відображати оновлені дані в публічному профілі.

1. Діаграма прецедентів (Use Case Diagram)

Переглянути PlantUML код 
@startuml
left to right direction
skinparam packageStyle rectangle

actor "Користувач" as User
actor "Креатор" as Creator
actor "Адміністратор" as Admin

' Наслідування акторів
User <|-- Creator
User <|-- Admin

rectangle "Система пошуку креаторів" {
    ' Основні прецеденти для Користувача
    User --> (Реєстрація)
    User --> (Авторизація)
    User --> (Пошук креаторів)
    
    ' Прецеденти для Креатора
    Creator --> (Додавання робіт)
    Creator --> (Редагування контенту)
    
    ' Прецеденти для Адміна
    Admin --> (Блокування користувача)

    ' Зв'язки Include (Включення)
    (Пошук креаторів) ..> (Фільтрація результатів) : <>
    (Авторизація) ..> (Перевірка даних) : <>
    (Реєстрація) ..> (Перевірка унікальності email) : <>
    (Редагування контенту) ..> (Збереження змін) : <>

    ' Зв'язок Extend (Розширення)
    (Блокування користувача) ..> (Авторизація) : <>
}
@enduml

2. Діаграма класів (Class Diagram)

Переглянути PlantUML код 
@startuml
skinparam classAttributeIconSize 0
skinparam monochrome true
skinparam shadowing false

' Базовий клас
class User {
  - id: Integer
  - username: String
  - email: String
  - passwordHash: String
  + register(): Boolean
  + login(): Boolean
  + logout(): void
  + createBoard(name: String): Board
}

' Наслідування (Спадкоємці)
class Creator {
  - isVerified: Boolean
  + addWork(w: Work): void
  + editWork(workId: Integer): void
  + deleteWork(workId: Integer): void
}

class Admin {
  - adminLevel: Integer
  + blockUser(userId: Integer): void
  + moderateContent(workId: Integer): Boolean
  + viewStatistics(): String
}

' Компоненти системи
class Profile {
  - profileId: Integer
  - description: String
  - skills: List
  - contactData: String
  + updateProfile(newData: String): Boolean
  + getDetails(): String
}

class Portfolio {
  - portfolioId: Integer
  - totalViews: Integer
  + getWorks(): List
}

class Work {
  - workId: Integer
  - title: String
  - description: String
  - imageUrl: String
  - tags: List
  + getDetails(): String
  + updateInfo(data: String): void
}

class Board {
  - boardId: Integer
  - name: String
  - createdAt: Date
  + addResource(work: Work): void
  + removeResource(workId: Integer): void
}

' Зв'язки (Relationships) та їх кратність

' Наслідування
User <|-- Creator : Наслідування
User <|-- Admin : Наслідування

' Композиція (сувора залежність, частина не існує без цілого)
Creator "1" *-- "1" Profile : Композиція
Creator "1" *-- "1" Portfolio : Композиція
User "1" *-- "0..*" Board : Композиція (Користувач має дошки)

' Агрегація (слабка залежність, елементи можуть існувати окремо)
Portfolio "1" o-- "0..*" Work : Агрегація (Портфоліо містить роботи)
Board "0..*" o-- "0..*" Work : Агрегація (Дошка зберігає роботи)

@enduml

3. Діаграма послідовності (Sequence Diagram)

Переглянути PlantUML код 
@startuml
title Діаграма послідовності: Додавання роботи до портфоліо (FR-05)

actor Creator as "Креатор"
participant UI as "Інтерфейс користувача"
participant Auth as "Сервіс авторизації"
participant PC as "PortfolioController"
database DB as "База даних"

autonumber

Creator -> UI : Натиснути "Додати роботу"
activate UI

UI -> Creator : Відобразити форму (файл, назва, опис)
Creator -> UI : Заповнити дані та натиснути "Зберегти"

UI -> Auth : Перевірити сесію (token)
activate Auth
Auth --> UI : Сесія активна
deactivate Auth

UI -> PC : uploadWork(workData)
activate PC

PC -> PC : Валідація даних (розмір файлу, тип)

alt #LightGreen Дані коректні
    PC -> DB : saveNewWork(CreatorID, Data)
    activate DB
    DB --> PC : WorkID, SuccessStatus
    deactivate DB
    
    PC --> UI : confirmSuccess()
    UI --> Creator : Повідомлення "Роботу успішно додано"
else #Pink Помилка валідації або доступу
    PC --> UI : errorResponse(errorMessage)
    deactivate PC
    UI --> Creator : Повідомлення про помилку
end

deactivate UI
@enduml
